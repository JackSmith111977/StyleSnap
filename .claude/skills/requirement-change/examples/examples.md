# 使用示例

## 示例 1：添加全新的功能需求

### 触发
用户："添加一个用户收藏夹功能，让用户可以收藏喜欢的风格"

### 第一轮询问
```
📋 已收到需求变更请求

## 第一轮询问（1/5）

1. 功能名称确认为"用户收藏夹"吗？
2. 用户收藏后，收藏计数需要公开显示吗？
3. 一个用户可以收藏多少个风格？有上限吗？
4. 收藏功能需要登录才能使用吗？
5. 建议优先级？（P0 核心/P1 重要/P2 可选）
```

### 第二轮询问
```
📋 收到回复

## 第二轮询问（2/5）

1. 需要创建 favorites 数据库表吗？还是用 JSON 字段？
2. 收藏后需要实时更新计数吗？
3. 需要"取消收藏"功能吗？
4. 用户能看到自己收藏的列表吗？（个人中心页面）
5. 有什么明确不做的范围？
```

### 影响分析输出
```markdown
## 影响分析报告

### 直接影响
- PRD.md：P1 章节新增"用户收藏"功能
- 数据库：需要 favorites 表 (user_id, style_id, created_at)
- API：需要 createFavorite, removeFavorite Server Actions
- 前端：需要收藏按钮组件、收藏列表页面

### 风险评估
- 技术风险：低（标准功能）
- 时间风险：中（需 2-3 天）
- 依赖风险：低（依赖用户系统，已完成）

请确认以上分析。
```

### 执行变更
```
✅ 已更新 PRD.md - 添加 F1.9 用户收藏功能
✅ 已更新 IMPLEMENTATION_PLAN.md - P1 阶段添加数据库设计
✅ 已生成变更记录

建议提交信息：
feat: 添加用户收藏功能需求

- 新增 PRD F1.9 用户收藏
- 更新 P1 阶段数据库设计
```

## 最佳实践

### 好的做法

✅ 询问前读取所有相关文档
✅ 每轮询问限 5 个问题
✅ 用表格呈现变更对比
✅ 明确标注"不做范围"
✅ 变更完成后生成提交草稿

### 避免的做法

❌ 假设用户需求
❌ 一次性问太多问题
❌ 不做影响分析直接变更
❌ 忘记更新 progress.txt
❌ 不生成变更记录
