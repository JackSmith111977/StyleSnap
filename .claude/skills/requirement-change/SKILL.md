# 需求变更 Skill

## 用途

当用户需要添加新功能需求、修改已有需求或调整优先级时，自动使用此 Skill。

**核心原则**：
1. **询问优先**：不做任何前提假设，通过多轮询问澄清需求
2. **影响分析**：评估变更对所有相关文档的影响
3. **同步更新**：确保 PRD.md、IMPLEMENTATION_PLAN.md、progress.txt 一致
4. **版本追踪**：每次变更生成可追溯的记录
5. **用户确认**：关键步骤需用户确认后再执行

**适用场景**：
- 添加新功能到 PRD
- 修改需求优先级（如 P1 → P0）
- 调整需求描述/范围
- 标记需求完成状态

**不适用场景**：
- 简单的文档拼写错误修正
- 代码层面的功能实现（直接 coding 即可）

## 工作流程

### 阶段 1：需求澄清（无限询问）

**步骤 1.1：读取当前文档状态**

```
1. 读取 PRD.md → 获取现有需求列表
2. 读取 IMPLEMENTATION_PLAN.md → 获取阶段划分
3. 读取 .claude/progress.txt → 获取当前进度
```

**步骤 1.2：执行无限询问**

**不做任何假设**，通过询问澄清以下问题（根据情况调整）：

**添加新需求时**：
```markdown
## 需求澄清 - 第 N 轮询问

请回答以下问题：

### 基础信息
1. 功能名称是什么？
2. 功能描述/用户故事是什么？（作为 XX 用户，我希望 XX，以便 XX）
3. 建议的优先级？（P0/P1/P2）

### 范围界定
4. 此功能涉及哪些模块？（前端/后端/数据库）
5. 是否需要新的数据库表/字段？
6. 是否需要新的 API/Server Action？

### 依赖关系
7. 此功能依赖其他功能吗？
8. 其他功能依赖此功能吗？

### 验收标准
9. 如何判断此功能完成？
10. 有什么明确的不做范围（Out of Scope）吗？
```

**修改优先级时**：
```markdown
## 变更澄清

1. 为什么需要调整优先级？
2. 此调整影响哪些其他需求？
3. 是否需要调整实现计划的时间安排？
```

**询问规则**：
- 每次最多 5 个问题
- 用户回答后评估是否足够清晰
- 如不清晰，继续下一轮询问
- 如已清晰，进入阶段 2

**等待用户确认需求清晰后，进入阶段 2。**

---

### 阶段 2：影响分析

**步骤 2.1：分析文档影响**

| 文档 | 检查项 |
|------|--------|
| PRD.md | 是否需要添加/修改功能条目 |
| IMPLEMENTATION_PLAN.md | 阶段划分、时间规划是否需调整 |
| progress.txt | 当前任务是否需更新 |
| CLAUDE.md | 规则是否需调整 |

**步骤 2.2：生成影响报告**

```markdown
## 影响分析报告

### 直接影响
- PRD.md：新增/修改 [X] 个功能条目
- IMPLEMENTATION_PLAN.md：调整 [X] 阶段

### 间接影响
- 数据库：需要/不需要 新表
- API：需要/不需要 新端点
- 前端：需要/不需要 新组件

### 风险评估
- 技术风险：[低/中/高]
- 时间风险：[低/中/高]
```

**输出报告，等待用户确认。**

---

### 阶段 3：执行变更

**步骤 3.1：更新 PRD.md**

根据需求类型执行：

**添加功能**：
```markdown
在对应优先级章节添加：
| [ID] | [功能名称] | [描述] |
```

**修改优先级**：
```markdown
1. 从原优先级章节移除
2. 添加到新优先级章节
```

**步骤 3.2：更新 IMPLEMENTATION_PLAN.md**

同步调整阶段划分和时间规划。

**步骤 3.3：更新 progress.txt**

如影响当前任务，更新进度状态。

**步骤 3.4：生成变更记录**

```markdown
## 变更记录 - 2026-XX-XX

| 变更项 | 变更前 | 变更后 | 原因 |
|--------|--------|--------|------|
| ... | ... | ... | ... |
```

---

### 阶段 4：确认与提交

**步骤 4.1：输出变更摘要**

```markdown
## 变更执行完成

### 已更新文件
- ✅ PRD.md
- ✅ IMPLEMENTATION_PLAN.md
- ✅ progress.txt

### 变更记录
[变更摘要]

### 建议提交信息
fix: 添加 [功能名称] 需求

- 新增 PRD 功能条目 [ID]
- 更新阶段划分
```

**步骤 4.2：等待用户确认提交**

询问：
```
是否执行 git 提交？
- 是：执行 git add → git commit
- 否：保持工作区状态
```

## 询问清单

### 新需求询问模板

每轮限 5 问，根据情况选择：

**第一轮（基础信息）**：
1. 功能名称是什么？
2. 一句话描述功能价值
3. 目标用户是谁？
4. 建议优先级（P0/P1/P2）
5. 预计工作量（天）

**第二轮（技术范围）**：
1. 需要新数据库表吗？
2. 需要新 API 吗？
3. 前端需要新页面吗？
4. 依赖现有功能吗？
5. 有什么明确不做的范围？

**第三轮（验收标准）**：
1. 如何测试此功能？
2. 有什么边界情况？
3. 性能要求？
4. 安全考虑？
5. 其他补充？

---

## 注意事项

### 边界情况

| 情况 | 处理方式 |
|------|----------|
| 需求模糊 | 继续询问，不假设 |
| 需求冲突 | 指出冲突，让用户决策 |
| 跨阶段调整 | 警告影响，需明确确认 |
| P0 级变更 | 必须用户明确确认 |

### 不做假设原则

**错误做法**：
- "我假设这个功能需要数据库支持..."
- "这个应该放在 P1 阶段..."

**正确做法**：
- "此功能需要数据库支持吗？"
- "你建议将此功能放在哪个优先级？"

## 示例

### 示例 1：添加新需求

**用户**："添加一个用户收藏功能"

**Skill 响应**：
```
📋 已收到需求变更请求

## 第一轮询问（1/5）

1. 功能名称是"用户收藏"吗？或有其他名称？
2. 用户收藏的具体场景是什么？（收藏风格/收藏组件？）
3. 建议优先级？（P0 核心/P1 重要/P2 可选）
4. 是否需要收藏分类/标签功能？
5. 用户收藏需要公开/私有选项吗？
```

**用户回答后**：
```
📋 收到回复

## 第二轮询问（2/5）

1. 需要创建 favorites 数据库表吗？
2. 一个用户可以收藏多少个风格？有上限吗？
3. 收藏后需要通知风格作者吗？
...
```

**需求清晰后**：
```
## 影响分析报告

### 直接影响
- PRD.md：P1 章节新增"用户收藏"功能
- 数据库：需要 favorites 表
...

请确认以上分析，或提出修改意见。
```

---

### 示例 2：调整优先级

**用户**："把社交功能优先级改为 P0"

**Skill 响应**：
```
📋 已收到优先级调整请求

当前状态：
- 社交功能（点赞/评论/收藏）：P1 阶段

## 变更澄清

1. 为什么需要调整为 P0？
2. 此调整影响 P0 其他功能吗？
3. 是否接受延长 MVP 发布时间？
```
