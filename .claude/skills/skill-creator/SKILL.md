# Skill Creator - Skill 创建器

## 用途

当用户想要创建新的 Skill 或修改已有 Skill 时，自动使用此 Skill。

**适用场景：**
- 创建全新 Skill（从 0 到 1）
- 修改已有 Skill（读取 → 修改 → 保存）
- 学习如何编写 Skill
- 获取 Skill 类型建议和最佳实践
- 优化已有 Skill 结构

**不适用场景：**
- 简单的 Skill 使用问题（直接回答即可）
- Skill 加载/触发故障排查

---

## 核心原则

1. **引导而非假设**：通过问答收集需求，不做任何假设
2. **推荐而非强加**：根据用户需求推荐最合适的 Skill 类型
3. **模板而非空壳**：提供 9 种类型的完整模板供选择
4. **检查而非直出**：生成后必须执行 Review 检查
5. **配置而非硬编码**：默认路径可配置

---

## 工作流程

### 阶段 1：需求澄清（问答引导）

**步骤 1.1：确认核心需求**

首先询问用户：

```markdown
## Skill 创建/优化需求确认

请回答以下问题，帮助我为你设计最合适的 Skill：

### 1. 这个 Skill 的主要用途是什么？
（例如：代码审查、文档生成、数据查询、部署流程等）

### 2. 你希望如何触发这个 Skill？
- 显式命令（如 /code-review）
- 自然语言（如"帮我检查一下代码"）
- 两者都需要

### 3. 这个 Skill 的主要使用者是谁？
- 仅自己使用
- 团队共享

### 4. 是否需要执行脚本或调用外部工具？
- 是（请说明需要什么工具）
- 否

### 5. 期望的输出格式是什么？
（例如：Markdown 报告、代码文件、JSON 数据等）
```

**步骤 1.2：推荐 Skill 类型**

根据用户回答，推荐最匹配的 Skill 类型（9 种之一）：

| 用户描述 | 推荐类型 |
|----------|----------|
| "教 Claude 使用内部库/API" | 库和 API 参考 |
| "测试/验证代码是否正确" | 产品验证 |
| "查询数据/生成报表" | 数据获取与分析 |
| "根据 XX 生成代码" | 代码生成 |
| "生成文档/报告" | 文档生成 |
| "检查代码质量/安全" | 代码审查 |
| "部署/发布流程" | 部署流程 |
| "统一编码风格/规范" | 团队规范 |
| "调研某个主题" | 调研整理 |

**步骤 1.3：展示 9 种 Skill 类型详情**

向用户展示完整的 9 种类型说明（详见 `references/skill-types.md`），供用户确认或更改。

**等待用户确认类型后，进入阶段 2。**

---

### 阶段 2：用例定义

**步骤 2.1：收集具体用例**

询问用户：

```markdown
## 用例定义

请描述 2-3 个这个 Skill 需要处理的具体用例：

**用例格式：**
- **触发条件**：用户说什么/做什么时触发
- **执行步骤**：Skill 需要执行的操作
- **预期结果**：输出什么/完成什么

**示例：**
1. **用例：审查 Pull Request**
   - 触发条件：用户提供 PR 链接或说"审查这个 PR"
   - 执行步骤：读取变更文件 → 执行安全检查清单 → 生成审查报告
   - 预期结果：包含问题汇总、具体问题位置、修复建议的 Markdown 报告
```

**步骤 2.2：确认触发方式**

```markdown
## 触发方式配置

### 显式触发（命令）
建议使用什么命令？
- 推荐：`/skill-name`（英文连字符，如 /code-review）

### 隐式触发（自然语言）
建议配置哪些触发词？
- 推荐：3-5 个中文触发词（如"审查代码"、"检查一下"）
```

**步骤 2.3：确认文件结构**

```markdown
## 文件结构配置

是否需要创建以下目录结构？

```
skill-name/
├── SKILL.md              # 必需：核心指令
├── templates/            # 可选：模板文件
├── references/           # 可选：参考文档
├── scripts/              # 可选：可执行脚本
└── checklists/           # 可选：检查清单
```

请确认需要哪些目录。
```

### 阶段 3：生成草稿

**步骤 3.1：加载对应类型模板**

根据用户选择的类型，加载对应的模板（详见 `templates/` 目录）。

**步骤 3.2：填充用户定制内容**

将用户提供的用例、触发方式等填入模板。

**步骤 3.3：输出草稿供预览**

```markdown
## Skill 草稿预览

```markdown
[生成的 SKILL.md 内容]
```

请确认以上草稿是否合适，或提出修改意见。
```

**等待用户确认后，进入阶段 4。**

---

### 阶段 4：写入文件

**步骤 4.1：确认输出位置**

```markdown
## 输出位置确认

默认路径：`.claude/skills/[skill-name]/`

当前工作目录：`[当前路径]`

请确认是否使用此路径，或指定其他路径。
```

**步骤 4.2：创建目录结构**

执行脚本或手动创建目录。

**步骤 4.3：写入文件**

使用 Write 工具写入 SKILL.md 及其他文件。

---

### 阶段 5：Review 检查

**步骤 5.1：执行检查清单**

加载 `checklists/review-checklist.md` 并逐项检查。

**步骤 5.2：输出检查结果**

```markdown
## Review 检查结果

### 格式检查
- [ ] YAML Frontmatter 格式正确
- [ ] 章节编号连续
- [ ] Markdown 格式正确

### 内容检查
- [ ] description 清晰具体（100 词内）
- [ ] 包含至少 2 个具体用例
- [ ] 包含工作流程
- [ ] 包含输出格式规范
- [ ] 包含注意事项

### 触发检查
- [ ] 配置了显式触发命令
- [ ] 配置了隐式触发词（3-5 个）

### 最佳实践检查
- [ ] 没有陈述显而易见的内容
- [ ] 包含可执行的检查清单
- [ ] 包含踩坑清单（如适用）
```

**步骤 5.3：修复问题**

如发现格式错误或内容缺失，主动修复。

---

### 阶段 6：测试建议

**步骤 6.1：生成测试用例**

```markdown
## 测试建议

创建 Skill 后，建议执行以下测试：

### 测试 1：显式触发
```
/[skill-name] 测试内容
```

### 测试 2：隐式触发
```
[触发词 1] 测试内容
```

### 测试 3：边界情况
```
[边界场景描述]
```

请执行以上测试，确认 Skill 正常工作。
```

**步骤 6.2：提供修改入口**

```markdown
如需修改 Skill，可以：
1. 直接告诉我需要修改的内容
2. 说"修改 [skill-name] Skill"
3. 手动编辑 `.claude/skills/[skill-name]/SKILL.md`
```

## 优化 Skill 结构流程

当用户要求优化已有 Skill 时（如"优化 research skill"）：

**步骤 O1：读取当前 Skill**

读取 `.claude/skills/[skill-name]/SKILL.md`

**步骤 O2：分析问题**

检查以下方面：
- 主文档是否过长（> 500 行）
- 是否包含重复内容
- 是否有可移至 references/的内容
- 是否符合标准目录结构

**步骤 O3：生成优化方案**

```markdown
## 优化方案

### 当前问题
- 主文档过长：[X] 行
- 重复内容：[具体内容]
- 缺少结构：[缺少的部分]

### 优化计划
1. 精简主文档至 500 行内
2. 移动 [内容] 至 references/
3. 创建 checklists/ 目录
4. 创建 examples/ 目录

### 优化后结构
```
skill-name/
├── SKILL.md（精简版主文档）
├── references/（参考资料）
├── checklists/（检查清单）
└── examples/（示例）
```

请确认优化方案。
```

**步骤 O4：执行优化**

按确认的方案执行优化。

## 输出格式规范

**阶段输出格式：**

每个阶段结束时，明确标注：

```markdown
---

**当前阶段：** 阶段 X - [阶段名称]
**下一步：** [下一步操作]
**等待确认：** [需要用户确认的内容]
```

**Skill 草稿格式：**

```yaml
---
name: [skill-name]
description: [清晰具体的描述，100 词内]
aliases: [alias1, alias2]
triggers: [触发词 1, 触发词 2, 触发词 3]
author: [用户名]
version: 1.0.0
metadata:
  category: [类别]
  type: [类型]
---

# [Skill 名称]

## 用途

[适用场景]

## 工作流程

[具体步骤]

## 输出格式

[输出规范]

## 注意事项

[边界情况、常见错误]
```

## 注意事项

### 安全与合规

- 不要创建需要敏感权限的 Skill（如读取 .env）
- 执行脚本前需用户确认
- 外部 API 调用需验证 SSL

### 性能考虑

- Skill 文件不宜过大（SKILL.md < 500 行）
- 复杂逻辑放入 resources/ 按需加载
- 避免在 Skill 中包含大量静态文本

### 限制说明

- 无法创建需要 API Key 的 Skill（除非用户提供）
- 无法访问需要登录的 Web 资源
- 视频内容只能获取简介
