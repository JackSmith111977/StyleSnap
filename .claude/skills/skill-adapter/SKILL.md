# skill-adapter - Skill 迁移适配向导

## 用途

当用户将 Skill 从其他项目迁移到当前项目时，自动扫描 Skill 文件，识别需要适配的配置项，并通过交互式询问引导用户完成所有适配。

**核心原则：**
1. 不假设业务：所有适配项通过询问确认，不做任何假设
2. 无限询问模式：持续询问直到所有适配项确认完成
3. 扫描优先：先扫描识别适配项，再逐项处理
4. 状态追踪：记录适配进度，支持中断后继续

---

## 工作流程

### 阶段 1：扫描适配项

**步骤 1.1：确认扫描范围**

询问用户：

```markdown
## 确认扫描范围

请告诉我需要适配的 Skill：

1. **单个 Skill**：提供 Skill 名称（如 `kb-trigger`）
2. **全部 Skill**：扫描 `.claude/skills/` 目录下所有 Skill
3. **指定目录**：提供具体路径

请回复选项或提供路径。
```

**步骤 1.2：扫描 Skill 文件**

读取以下文件识别适配项：
- `SKILL.md` - 主文件（必需）
- `references/` - 参考文档
- `scripts/` - 脚本文件
- `checklists/` - 检查清单

**步骤 1.3：识别适配项类型**

| 适配项类型 | 识别模式 | 示例 |
|------------|----------|------|
| **硬编码路径** | 绝对路径或特定目录名 | `Knowledge Base/`、`/Users/xxx/` |
| **脚本依赖** | `.claude/scripts/` 引用 | `node .claude/scripts/xxx.js` |
| **Hook 配置** | `settings.local.json` 引用 | `UserPromptSubmit` hooks |
| **领域特定** | 技术名词库、模式库 | `TECH_ENTITIES`、`COMMAND_PATTERNS` |
| **文件引用** | 相对路径引用 | `references/xxx.md` |

---

### 阶段 2：列出适配项

**步骤 2.1：汇总适配项**

输出格式：

```markdown
## 扫描结果

发现以下需要适配的项目：

### Skill: [Skill 名称]

| 序号 | 适配项类型 | 位置 | 当前值 | 优先级 |
|------|------------|------|--------|--------|
| 1 | 硬编码路径 | SKILL.md:56 | `Knowledge Base/` | 高 |
| 2 | 脚本依赖 | SKILL.md:37 | `.claude/scripts/kb-checker.js` | 高 |
| 3 | 领域特定 | scripts/xxx.js:20 | `TECH_ENTITIES` | 中 |

**待适配总数：** 3 项
```

**步骤 2.2：确认适配顺序**

询问用户：

```markdown
## 确认适配顺序

建议按以下顺序处理：

1. **高优先级**：路径和脚本依赖（必须先处理）
2. **中优先级**：领域特定配置
3. **低优先级**：可选优化项

是否按建议顺序处理？或指定其他顺序。
```

---

### 阶段 3：逐项询问适配

**步骤 3.1：逐项询问**

对每个适配项，输出：

```markdown
## 适配项 #1：硬编码路径

**位置：** `SKILL.md:56`
**当前值：** `Knowledge Base/[主题]/`
**说明：** 这是知识库文档存储路径

**请选择适配方式：**

1. **使用默认路径**：`Knowledge Base/`（当前项目目录）
2. **自定义路径**：输入新路径
3. **跳过**：暂不处理，后续手动修改

请回复选项编号或输入路径。
```

**步骤 3.2：执行修改**

根据用户选择：
- 选项 1：使用默认路径
- 选项 2：读取用户输入的路径并替换
- 选项 3：标记为"待处理"，继续下一项

**步骤 3.3：确认修改**

```markdown
已将 `Knowledge Base/` 替换为 `[用户选择的路径]`

**修改预览：**
```diff
- 存储位置：Knowledge Base/[主题]/[主题名] 核心知识体系.md
+ 存储位置：Docs/[主题]/[主题名] 核心知识体系.md
```

是否确认此修改？
- 回复「是」继续下一项
- 回复「否」撤销并重试
```

---

### 阶段 4：验证适配完成

**步骤 4.1：检查适配状态**

扫描所有适配项，确认状态：

| 状态 | 说明 |
|------|------|
| ✅ 已完成 | 用户确认并修改完成 |
| ⏳ 待处理 | 用户选择跳过 |
| ❌ 失败 | 修改执行失败 |

**步骤 4.2：生成适配报告**

```markdown
## 适配完成报告

**Skill 名称：** kb-trigger
**适配时间：** 2026-03-25 10:30
**适配结果：**

| 适配项 | 状态 | 修改内容 |
|--------|------|----------|
| 硬编码路径 | ✅ 完成 | `Knowledge Base/` → `Docs/` |
| 脚本依赖 | ✅ 完成 | 脚本路径已更新 |
| 领域特定 | ⏳ 待处理 | 用户选择跳过 |

**待处理项：** 1 项
**建议：** 后续手动处理待处理项
```

## 踩坑清单

| 陷阱 | 错误做法 | 正确做法 |
|------|----------|----------|
| **假设路径** | 直接使用默认路径 | 必须询问用户确认 |
| **批量修改** | 一次性修改所有项 | 逐项询问，逐项确认 |
| **遗漏检查** | 只检查 SKILL.md | 扫描所有相关文件 |
| **状态丢失** | 不记录适配进度 | 写入状态文件支持继续 |
| **跳过不管** | 跳过项不记录 | 记录待处理项并提醒 |

## 注意事项

### 交互原则
- **不假设**：所有适配项必须询问用户
- **不跳过**：跳过的项必须记录并提醒
- **不丢失**：适配状态必须保存

### 文件安全
- 修改前先备份原文件
- 每次修改后确认
- 支持撤销操作

### 限制说明
- 无法自动检测业务逻辑相关的适配项
- 无法访问需要权限的文件
