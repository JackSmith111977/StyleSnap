# project-start - 文档优先的 Agent 智能开发启动

## 用途

当用户需要初始化一个新项目时，通过**无限询问模式**将空项目完整初始化，包括：
1. `.claude/` 标准目录配置（CLAUDE.md、progress.txt 等）
2. 文档优先系统配置（PRD、APP_FLOW、TECH_STACK 等）
3. Agent 对话规约、上下文和对话模式定义

**核心原则：**
1. **不假设任何前提**：所有配置通过询问确认
2. **无限询问模式**：直到所有前提清晰才创建文档
3. **文档优先**：在编码之前建立完整的文档基础
4. **分阶段执行**：每阶段确认后再进入下一阶段

---

## 工作流程

### 阶段 0：启动与项目类型确认

**步骤 0.1：欢迎与范围说明**

```markdown
🚀 **项目启动向导**

我将通过无限询问模式帮你初始化新项目，包括：
1. .claude/目录配置（CLAUDE.md、progress.txt、Agent 规约等）
2. 文档优先系统（PRD、APP_FLOW、TECH_STACK 等）
3. Agent 对话规约和上下文定义

**询问原则：** 不假设任何前提，直到你确认所有配置。

让我们开始。
```

**步骤 0.2：项目基本信息**

询问：

```markdown
## 阶段 0：项目基础信息

请回答以下问题：

### 1. 项目名称
[输入项目名]

### 2. 项目类型
- Web 前端（React/Vue/Angular 等）
- Web 后端（Node.js/Python/Go 等）
- 全栈应用
- 移动端应用（React Native/Flutter 等）
- 库/框架
- 其他（请描述）

### 3. 团队规模
- 个人项目
- 小团队（2-10 人）
- 企业项目（10 人+）

### 4. 是否需要项目模板
- 是（推荐什么模板？）
- 否（完全自定义）
```

**等待用户回答后，进入阶段 1。**

---

### 阶段 1：.claude 目录初始化

**步骤 1.1：CLAUDE.md 内容询问**

逐项询问：

```markdown
## 阶段 1：.claude 目录初始化

### 1.1 项目概述

**问题：**
- 用一句话描述项目的核心目标
- 项目要解决什么问题
- 目标用户是谁

---

### 1.2 技术栈

**问题：**
- 前端框架（如 React 18、Vue 3）
- 后端框架（如 Node.js + Express、Python + FastAPI）
- 数据库（如 PostgreSQL、MongoDB）
- 云服务/部署平台（如 Vercel、AWS）
- 其他关键技术（如 GraphQL、Redis）

---

### 1.3 代码规范

**问题：**
- 主要编程语言
- 代码风格约定（缩进、命名、导入顺序等）
- 是否使用 Linter（ESLint、Prettier 等）
- 提交信息规范（如 Conventional Commits）

---

### 1.4 开发命令

**问题：**
- 依赖安装命令
- 开发启动命令
- 构建命令
- 测试命令
- 其他常用脚本

---

### 1.5 Agent 对话规约

**问题：**
- 偏好什么对话模式（简洁/详细/分步骤）
- 是否需要特定的上下文管理方式
- 代码生成时的约束（如"先生成接口再实现"）
- 审查代码时的重点（如"优先关注安全性"）

---

### 1.6 重要约定

**问题：**
- 项目特定的开发约定
- 团队内部的工作流程
- 其他需要 Claude 长期记住的信息
```

**步骤 1.2：逐项确认**

每询问完一个子项，输出摘要供确认：

```markdown
### 确认：CLAUDE.md - [子项名称]

基于你的回答，我将记录：

**[子项名称]：**
- [要点 1]
- [要点 2]
- ...

请确认：
- 回复「是」继续下一项
- 回复「否」修改内容
- 回复「跳过」暂不处理
```

**步骤 1.3：其他.claude 文件**

```markdown
### 1.7 progress.txt 配置

**结构：**
# 项目进度追踪

## 项目信息
- 名称：[项目名]
- 类型：[类型]
- 创建：[日期]

## 状态
- 已完成：[]
- 进行中：[]
- 接下来：[]

## 当前阶段：[阶段名称]

是否采用此结构？或需要调整？

---

### 1.8 settings.local.json

**询问：**
- 是否需要配置特定的权限
- 是否需要配置 MCP 服务
- 是否需要配置 Hooks

---

### 1.9 agents/ 目录

**询问：**
- 是否需要预定义子代理（如 code-reviewer、debugger）
- 如需，请描述每个代理的职责

---

### 1.10 rules/ 目录

**询问：**
- 是否需要模块化的规则文件
- 如需，请描述规则类别（如 code-style.md、testing.md）
```

**等待所有子项确认后，进入阶段 2。**

---

### 阶段 2：文档优先系统初始化

**步骤 2.1：说明文档结构**

```markdown
## 阶段 2：文档优先系统

我将创建以下文档结构：

```
docs/
├── main/              # 核心文档
│   ├── PRD.md                    # 产品需求文档
│   ├── APP_FLOW.md               # 应用流程/架构
│   ├── TECH_STACK.md             # 技术栈详情
│   ├── FRONTEND_GUIDELINES.md    # 前端规范
│   ├── BACKEND_STRUCTURE.md      # 后端结构
│   └── IMPLEMENTATION_PLAN.md    # 实施计划
├── research/          # 调研报告
├── knowledge-base/    # 技术栈知识库、指南
└── reference/         # API 文档（供 Agent 引用）
```

是否确认此结构？或需要调整？
```

**步骤 2.2：逐项询问文档内容**

对每个文档进行无限询问：

### PRD.md（产品需求文档）

```markdown
### 2.2.1 PRD.md - 产品需求文档

**询问列表：**
1. 产品愿景：产品要实现的长远目标是什么
2. 问题陈述：当前存在什么问题需要解决
3. 目标用户：谁会使用这个产品，用户画像是什么
4. 核心功能：必须实现的功能列表（MoSCoW 优先级）
5. 非功能需求：性能、安全、可用性等要求
6. 成功指标：如何衡量产品成功（KPI、OKR）
7. 约束条件：时间、预算、技术等限制
8. 依赖关系：外部依赖、第三方服务

**询问方式：**
逐个问题询问，每个问题后确认理解是否正确：

"关于 [问题]，我的理解是：[复述]。是否正确？"
```

### APP_FLOW.md（应用流程）

```markdown
### 2.2.2 APP_FLOW.md - 应用流程/架构

**询问列表：**
1. 用户流程：用户从进入到完成目标的路径
2. 系统架构：整体架构图（单体/微服务/Serverless）
3. 数据流：数据如何在系统中流动
4. 核心模块：系统的主要组成部分
5. 接口设计：内部和外部接口定义
6. 状态管理：应用状态如何管理
7. 路由结构：页面/视图的组织方式
```

### TECH_STACK.md（技术栈）

```markdown
### 2.2.3 TECH_STACK.md - 技术栈详情

**询问列表：**
1. 前端技术栈：框架、UI 库、状态管理、构建工具
2. 后端技术栈：框架、ORM、认证、API 规范
3. 数据库：类型、版本、设计原则
4. 云服务：部署平台、CDN、存储
5. 第三方服务：支付、邮件、短信等
6. 开发工具：IDE、调试、测试工具
7. 版本管理：Git 流程、分支策略
```

### FRONTEND_GUIDELINES.md（前端规范）

```markdown
### 2.2.4 FRONTEND_GUIDELINES.md - 前端规范

**询问列表：**
1. 目录结构：前端代码的组织方式
2. 组件规范：组件设计原则、命名、props 定义
3. 样式规范：CSS 方案、命名约定、响应式
4. 状态管理：全局状态、本地状态、缓存策略
5. API 调用：请求封装、错误处理、loading 状态
6. 性能优化：懒加载、代码分割、缓存
7. 可访问性：a11y 要求
```

### BACKEND_STRUCTURE.md（后端结构）

```markdown
### 2.2.5 BACKEND_STRUCTURE.md - 后端结构

**询问列表：**
1. 目录结构：后端代码的组织方式
2. 路由设计：URL 规范、版本控制
3. 控制器规范：请求处理、响应格式
4. 数据模型：实体设计、关系定义
5. 服务层：业务逻辑组织
6. 认证授权：用户认证、权限控制
7. 错误处理：错误码、异常处理
8. 日志策略：日志级别、格式、存储
```

### IMPLEMENTATION_PLAN.md（实施计划）

```markdown
### 2.2.6 IMPLEMENTATION_PLAN.md - 实施计划

**询问列表：**
1. 里程碑：关键交付节点
2. 迭代计划：Sprint 划分、每轮目标
3. 任务分解：功能拆解为具体任务
4. 优先级：任务优先级排序
5. 时间估算：每项任务的时间预估
6. 风险管理：潜在风险和应对方案
```

**步骤 2.3：其他文档目录说明**

## 踩坑清单

| 陷阱 | 错误做法 | 正确做法 |
|------|----------|----------|
| **假设技术栈** | "我猜你用的是 React" | "请告诉我你使用的前端框架" |
| **跳过确认** | 直接生成文档 | 每项配置后询问确认 |
| **遗漏 Agent 规约** | 只配置基础信息 | 必须包含对话规约和上下文 |
| **文档空洞** | 只创建空文件 | 通过询问填充实质内容 |
| **不检测联动** | 忽略已有 Skill | 检测并提示 skill-adapter |

## 注意事项

### 无限询问原则
- 不假设任何前提
- 每项配置必须确认
- 用户说"跳过"才能跳过
- 问题数量取决于项目复杂度

### 文件安全
- 创建前确认文件不存在或覆盖
- 支持撤销操作
- 保留配置历史

### 联动触发
- 检测技能迁移时主动提示 skill-adapter
- 检测 MCP 需求时主动提示配置
