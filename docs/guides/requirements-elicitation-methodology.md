## 文档目的

本文档总结了一套**通过结构化提问梳理需求并编写应用流程文档**的方法论，适用于：
- 产品经理梳理产品需求
- 开发者编写技术文档
- 团队协作对齐理解
- 个人项目结构化思考

---

## 方法论核心：审问式需求梳理

### 核心理念

> **不要假设任何问题。问问题直到没有假设剩下。**

在 Planning 模式下，通过系统性、结构化的提问，将用户脑海中的模糊想法转化为清晰、可执行的需求文档。

## 第一阶段：准备阶段

### 1.1 确定文档类型

在开始提问前，先确认要编写的文档类型：

| 文档类型 | 用途 | 核心内容 |
|----------|------|----------|
| APP_FLOW.md | 应用流程文档 | 页面流程、用户导航路径、触发事件、异常流、屏幕清单 |
| PRD.md | 产品需求文档 | 功能需求、优先级、用户故事、验收标准 |
| TECH_STACK.md | 技术栈文档 | 技术选型、版本、配置、集成方式 |
| API_SPEC.md | API 规格文档 | 端点、请求/响应、错误码、认证方式 |

### 1.2 确定文档范围

向用户确认文档的**范围边界**：

- 哪些功能是 MVP 必须的？
- 哪些功能是后续迭代的？
- 文档的详细程度要求？

**示例提问**：
> "以下流程哪些是 MVP 必须的？
> - [ ] 游客浏览风格 → 查看代码
> - [ ] 用户注册/登录
> - [ ] 收藏风格
> - [ ] 发表评论"

## 第二阶段：提问阶段

### 2.1 提问原则

| 原则 | 说明 | 反例 |
|------|------|------|
| **每次仅 5 问** | 避免认知过载，便于用户逐一回答 | 一次性抛出 20 个问题 |
| **封闭式优先** | 用选项引导，降低回答成本 | "你觉得应该怎么设计？" |
| **提供默认值** | 给出推荐选项，用户可确认或修改 | 完全开放式问题 |
| **逐组深入** | 按主题分组，每组问完确认再继续 | 跳跃式提问 |
| **记录确认** | 每组结束后复述用户答案确认理解 | 不问清楚就继续 |

### 2.3 问题设计技巧

#### 技巧 1：提供选项而非开放提问

❌ 差问题：
> "用户认证怎么设计？"

✅ 好问题：
> "注册/登录方式有哪些？
> - [ ] 邮箱 + 密码（需要邮箱验证吗？）
> - [ ] GitHub OAuth
> - [ ] Google OAuth
> - [ ] 忘记密码流程需要吗？"

#### 技巧 2：提供推荐选项

❌ 差问题：
> " Session 要保持多久？"

✅ 好问题：
> "需要'记住我'功能吗？
> - [ ] 需要（Session 保持多久？7 天/30 天/永久）
> - [ ] 不需要"

#### 技巧 3：概念解释 + 提问

当涉及专业概念时，先解释再提问：

```markdown
## Q33 补充说明：SEO 配置项

### 1. SSR 静态生成
- **是什么**：风格详情页在服务端生成 HTML，而非纯客户端渲染
- **好处**：百度/Google 等搜索引擎能更好抓取内容
- **Next.js 实现**：使用 `generateStaticParams` + `ISR` 增量更新

### 2. sitemap.xml
- **是什么**：一个 XML 文件，列出网站所有页面，帮助搜索引擎发现内容
...

## 重新提问 Q33
### Q33. SEO 相关配置，以下哪些需要？
- [ ] SSR 静态生成
- [ ] sitemap.xml
- [ ] OG / Twitter Card
```

#### 技巧 4：调研后提问

当用户不确定时，先快速调研再提问：

```markdown
## Q44 补充说明：本地模拟器 vs 云开发环境

基于搜索结果，我来补充说明两者的区别：

### 方案 A：Supabase CLI 本地模拟器
**优点**：完全离线开发、无云服务费用、数据完全本地
**缺点**：需要 Docker 支持、初次设置稍复杂

### 方案 B：直接连接 Supabase 云开发项目
**优点**：设置简单、与云端一致、本地资源占用低
**缺点**：需要网络连接、免费项目有资源限制

## 重新提问 Q44
### Q44. 开发环境选择？
- [ ] 仅本地模拟器
- [ ] 仅云开发项目
- [ ] 混合模式（本地 + 云端，最佳实践）
```

## 第三阶段：整理阶段

### 3.1 答案记录格式

每组问答结束后，记录确认内容：

```markdown
**A1-5 确认：**
- 访客：可浏览所有风格、查看详情和代码
- 登录用户：可收藏、点赞、评论、提交风格、创建合集
- 管理员：集成在 `/admin/*` 路由
- MVP：包含所有列出的流程
- 首页：风格列表网格 + 搜索框 + 精选风格轮播
```

### 3.2 歧义消除

当问题存在歧义时，单独澄清：

```markdown
**关于 Q9 的说明：**
问题是关于**错误提示的展示方式**，例如：
- 用户输入已注册的邮箱时，错误信息显示在哪里？
  - [ ] 表单内联显示（字段下方红色文字）
  - [ ] Toast 弹窗提示
  - [ ] 两者结合
```

### 3.3 最终汇总

所有问题结束后，进行完整汇总确认：

```markdown
## 问题环节完成 - 信息汇总确认

### 用户角色与权限
- 访客：可浏览所有风格、查看详情和代码
- 登录用户：可收藏、点赞、评论、提交风格、创建合集
...

### 技术细节
- 图片：Supabase Storage，存原图，图片处理参数获取缩略图
- 部署：Vercel，环境变量管理，先默认域名
- 开发：混合模式（本地模拟器 + 云端联调）
- 数据：种子脚本初始化

---
如确认无误，我开始编写 APP_FLOW.md 文档。
```

## 4. 核心用户流程
- 流程 1：触发条件、前置条件、主成功流、业务异常流、系统异常流
- 流程 2：...

- 公共页面

- 流程图
- 术语表

### 4.2 流程描述格式

每个流程包含以下要素：

```markdown
### 4.x XXX 流程

**触发条件**：用户访问首页或风格列表页

**前置条件**：用户已登录且邮箱已验证

**主成功流**：
```
1. 用户访问 /styles/:id
2. 用户点击「收藏」按钮
3. 系统切换按钮状态
4. 系统发送 API 请求
5. 系统更新本地缓存
6. 显示 Toast 提示
7. 流程结束
```

**业务异常流**：
| 异常 | 触发条件 | 系统处理 |
|------|----------|----------|
| 未登录 | 步骤 2 | 跳转 /auth/login |
| 邮箱未验证 | 步骤 2 | 显示验证 Modal |

**系统异常流**：
| 异常 | 触发条件 | 系统处理 |
|------|----------|----------|
| 网络错误 | 步骤 4 | 按钮回滚状态，显示重试 |
| 服务器错误 | 步骤 4 | 显示系统繁忙提示 |
```

### 4.3 表格使用规范

| 场景 | 推荐表格类型 |
|------|--------------|
| 角色权限对比 | 矩阵表（行角色 × 列功能） |
| 路由列表 | 三列表（路径、名称、说明） |
| 异常处理 | 三列表（异常、触发条件、处理） |
| 屏幕清单 | 四列表（ID、名称、路由、元素） |

## 常见问题 FAQ

### Q1: 用户答案模糊怎么办？

**场景**：用户回答"都可以"或"你决定"

**处理**：
1. 提供推荐选项并说明理由
2. 确认用户是否接受
3. 记录决策原因

**示例**：
> "推荐方案 C（混合模式），理由：
> - 本地开发快速迭代
> - 云端联调确保一致性
> - 业界最佳实践
>
> 是否按此方案继续？"

### Q2: 用户中途修改之前的答案怎么办？

**处理**：
1. 接受修改，不质疑一致性
2. 更新答案记录
3. 询问是否影响已编写内容

**示例**：
> "收到，已更新 A15 的答案。
> 之前的问题 X 中提到 Y，是否需要一并调整？"

### Q3: 问题数量太多用户不耐烦怎么办？

**处理**：
1. 说明问题数量的必要性
2. 可分多次会话完成
3. 优先核心问题，次要问题按惯例

**示例**：
> "理解，共 45 问确实较多。
> 建议分 3 次完成：
> - 本次：Q1-20（核心流程）
> - 下次：Q21-35（用户功能）
> - 最后：Q36-45（技术细节）
>
> 或次要问题我按行业惯例填写，您确认即可。"

### Q4: 用户需求不在标准分组内怎么办？

**处理**：
1. 记录特殊需求
2. 追加针对性问题
3. 扩展问题分组框架
