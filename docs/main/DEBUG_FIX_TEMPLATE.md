## 修复流程概览

```
┌─────────────────────────────────────────────────────────────────┐
│                      StyleSnap 修复流程                          │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  1. 问题报告 → 2. 上下文收集 → 3. 根因分析                      │
│         ↓                  ↓                                    │
│  5. 文档归档 ← 4. 修复验证 ← ✦ 方案确认 (关键节点) ← 实施修复   │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

### 流程说明

| 阶段 | 产出物 | 适用问题级别 |
|------|--------|--------------|
| 1. 问题报告 | 问题描述、复现步骤、错误日志 | 所有问题 |
| 2. 上下文收集 | 日志、截图、影响范围 | P0/P1 必须 |
| 3. 根因分析 | 根因假设、验证结果 | P0/P1 必须 |
| ✦ **方案确认** | [`FIX_BEFORE_CONFIRM_CHECKLIST.md`](./FIX_BEFORE_CONFIRM_CHECKLIST.md) | **P0/P1 必须，P2 建议** |
| 4. 实施修复 | 代码变更、测试验证 | 所有问题 |
| 5. 文档归档 | [`DEBUG_FIX_REPORT_TEMPLATE.md`](./DEBUG_FIX_REPORT_TEMPLATE.md) | P0/P1 必须，P2 建议 |

### 问题级别定义

| 级别 | 定义 | 流程要求 | 例子 |
|------|------|----------|------|
| **P0 紧急** | 核心功能完全不可用，影响所有用户 | 完整流程 + 立即修复 | 登录失败、数据库连接中断 |
| **P1 重要** | 核心功能部分不可用，影响部分用户 | 完整流程 + 24h 内修复 | 收藏功能失效、支付失败 |
| **P2 一般** | 非核心功能问题，影响体验 | 简化流程 | 样式问题、轻微 UI 错位 |

## 流程详解

### 阶段 1: 问题报告

**输入**: 用户报告、测试发现、监控告警

**输出**:
- 问题描述（一句话）
- 复现步骤
- 错误日志/截图

**模板**:
```markdown
## 问题报告
- **问题**: [一句话描述]
- **复现步骤**: 1. ... 2. ... 3. ...
- **错误日志**: [粘贴日志或截图]
```

---

### 阶段 2: 上下文收集

**目的**: 收集足够信息以便准确定位问题

**检查清单**:
- [ ] 服务器日志已检查
- [ ] 客户端日志已检查
- [ ] 数据库查询已检查（如适用）
- [ ] 相关文件已定位

**工具**:
- Next.js MCP: `get_errors`, `get_logs`
- Playwright MCP: 浏览器自动化测试
- Supabase Dashboard: 数据库查询日志

---

### 阶段 3: 根因分析

**方法**:
1. 列出所有可能的根因假设
2. 逐一验证每个假设
3. 确认最终根因

**模板**:
```markdown
## 根因分析
| 假设 | 验证方法 | 结果 |
|------|----------|------|
| 假设 1 | ... | ✅ 确认 / ❌ 排除 |
| 假设 2 | ... | ✅ 确认 / ❌ 排除 |

**最终根因**: [一句话总结]
```

---

### ✦ 阶段 4: 方案确认 (关键节点)

**目的**: 在实施前确认修复方案，避免盲目修复

**适用**: P0/P1 问题**必须**执行，P2 问题**建议**执行

**步骤**:
1. 填写 [`FIX_BEFORE_CONFIRM_CHECKLIST.md`](./FIX_BEFORE_CONFIRM_CHECKLIST.md)
2. 列出至少 2 种修复方案并对比
3. 评估影响范围和回归测试清单
4. **用户/PM 确认方案** ← 关键确认点

**产出物**:
```markdown
## 修复方案对比
| 方案 | 优点 | 缺点 | 评估 |
|------|------|------|------|
| A | ... | ... | ✅ 推荐 |
| B | ... | ... | ❌ 不推荐 |

**选择理由**: ...
```

---

### 阶段 5: 实施修复

**原则**:
- 小步提交，频繁验证
- 优先保证现有功能不被破坏
- 添加必要的测试用例

**验证清单**:
```bash
pnpm typecheck  # 类型检查
pnpm lint       # 代码风格
pnpm build      # 构建验证
pnpm test       # 单元测试
```

**MCP 浏览器复测**:
- [ ] 导航到相关页面
- [ ] 执行关键操作
- [ ] 检查控制台无错误

---

### 阶段 6: 文档归档

**P0/P1 问题**: 必须创建修复报告

**模板**: [`DEBUG_FIX_REPORT_TEMPLATE.md`](./DEBUG_FIX_REPORT_TEMPLATE.md)

**内容包括**:
- 问题描述和影响范围
- 根因分析过程
- 修复方案对比和选择理由
- 经验教训和预防措施
- 相关文档链接
